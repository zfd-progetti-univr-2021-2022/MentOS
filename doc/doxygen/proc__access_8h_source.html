255 // == Memory clobbers =========================================================
256 // Memory clobber implies a fence, and it also impacts how the compiler treats
257 // potential data aliases. A memory clobber says that the asm block modifies
258 // memory that is not otherwise mentioned in the asm instructions.
259 // So, for example, a correct use of memory clobbers would be when using an
260 // instruction that clears a cache line. The compiler will assume that
261 // virtually any data may be aliased with the memory changed by that
262 // instruction. As a result, all required data used after the asm block
263 // will be reloaded from memory after the asm completes. This is much more
264 // expensive than the simple fence implied by the "volatile" attribute.
265 
266 // == Volatile Block ==========================================================
267 // Making an inline asm block "volatile" as in this example, ensures that,
268 // as it optimizes, the compiler does not move any instructions above or
269 // below the block of asm statements.
270 // ASM(" addic. %0,%1,%2\n" : "=r"(res): "=r"(a),"r"(a))
271 // This can be particularly important in cases when the code is accessing
272 // shared memory.
