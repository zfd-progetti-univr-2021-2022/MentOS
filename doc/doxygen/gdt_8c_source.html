129 //
130 // == VIRTUAL MEMORY SCHEMES ==================================================
131 // x86 supports two virtual memory schemes:
132 // segmentation (mandatory): managed using the segment table, GDT.
133 // paging (optional) : managed using the page table, PDT.
134 // Most operating systems want to to use paging and don't want the
135 // segmentation, but its mandatory and can't just be disabled.
136 //
137 // So the trick is to disable its effect as it wasn't there. This can usually
138 // be done by creating 4 large overlapped segments descriptors (beside the
139 // null segment):
140 // segment index 0 : null segment descriptor
141 // segment index 1 : CODE segment desc. for the privileged (kernel) mode
142 // segment index 2 : DATA segment desc. for the privileged (kernel) mode
143 // segment index 3 : CODE segment desc. for the non-privileged (user) mode
144 // segment index 4 : DATA segment desc. for the non-privileged (user) mode
145 //
146 // all these segments starts from 0x00000000 up to 0xffffffff, so you end
147 // up with overlapped large segments that is privileged code and data, and
148 // non-privileged code and data in the same time. This should open up the
149 // virtual memory and disable the segmentation effect.
150 //
151 // The processor uses the segment selectors (segment registers cs, ds, ss ...)
152 // to find out the right segment (once again, the segmentation is must).
153 //
154 // == SEGMENT SELECTOR ========================================================
155 // Every segment selector is 16 bit size and has the following layout (source):
156 // |15 3| 2|1 0|
157 // |----- Index (13-bit) ----- | TI | RPL |
158 // where TI is the Table Indicator:
159 // 0 - GDT
160 // 1 - LDT
161 // and RPL encodes in 2 bits the Requestor Privilege Level (RPL):
162 // 00 - Highest
163 // 01
164 // 10
165 // 11 - Lowest
166 // Regarding the `privilege level`, x86 supports 4 levels, but only two of them
167 // are actually used (00 highest, and 11 lowest).
168 //
169 // The remaining 13 bits indicates the segment index.
170 //
171 // == GDT_FLUSH ===============================================================
172 // If you look in gdt.S, you will see a `jmp 0x08` at the end of the gdt_flush.
173 // Now, if you interpret the 0x08 that is loaded in cs, it will be in binary:
174 // | 0000000000001| 0| 11|
175 // | index 3 (code) | GDT| privileged|
176 //
177 // and the 0x10 that is loaded in ds, ss, ... :
178 // | 0000000000010| 0| 11|
179 // | index 3 (code) | GDT| privileged|
180 //
181 // == SS of a USER MODE PROGRAM ===============================================
182 // If you read the segment selectors of any user mode program you should see
183 // that the cs value is 27 (0x1b) which means:
184 // | 0000000000011| 0| 11|
185 // | index 3 (code) | GDT| non-privileged|
186 // and the data selectors ds, ss, ..., should store 35 (0x23):
187 // | 0000000000100| 0| 11|
188 // | index 4 (data) | GDT| non-privileged|
189 //
